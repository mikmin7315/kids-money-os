# Kids Money OS MVP Spec

## 1. 왜 이런 구조로 설계했는가

이 서비스의 핵심은 달력이 아니라 `행동 -> 금융 이벤트 -> 리포트`의 연결입니다.

그래서 모델을 다음처럼 나눴습니다.

- 규칙 테이블: 부모가 설정하는 기준
- 로그/요청 테이블: 아이가 실제로 만든 이벤트
- 거래 테이블: 돈의 흐름을 원장처럼 누적 기록
- 요약 테이블: 월간 리포트와 대시보드 최적화

이 구조를 쓰면:

- 행동과 돈 흐름을 느슨하게 연결할 수 있고
- 추후 포인트, 미션, AI 리포트가 추가돼도 거래 원장은 유지되며
- 날짜 기반 조회와 월간 집계가 단순해집니다.

## 2. 주요 사용자 플로우

### 부모 플로우

1. 부모 로그인
2. 자녀 선택
3. 행동 규칙 생성
4. 용돈/이자/빌려쓰기 정책 설정
5. 요청 승인
6. 월간 리포트 확인

### 아이 플로우

1. 아이 모드 진입
2. 오늘 할 일 확인
3. 행동 완료 체크
4. 잔액/저축/빌린 돈 확인
5. 소비 기록 입력 또는 저축 이동
6. 필요 시 빌려쓰기 요청
7. 기록 화면에서 흐름 복습

## 3. 주요 화면 와이어프레임

### 홈 대시보드

```text
[헤더: 부모/아이 모드, 자녀 선택]
[핵심 요약 카드 4개]
  잔액 | 저축액
  빌린 돈 | 적용 이자율

[오늘의 행동]
  체크 가능한 행동 리스트

[최근 금융 활동]
  용돈/소비/저축/이자/상환 타임라인

[이번 달 요약]
  지급/소비/저축/행동 성공률

[하단 이동]
  홈 | 기록 | 승인 | 리포트 | 설정
```

### 기록 화면

```text
[월 선택]
[리스트/캘린더 토글]
[날짜별 그룹]
  4/03 행동 승인
  4/03 보상 지급
  4/05 소비
  4/07 저축
```

### 행동 관리 화면

```text
[규칙 목록]
  제목 / 보상 / 이자 영향 / 승인 필요 여부
[추천 규칙 카드]
[최근 행동 로그]
```

### 빌려쓰기 승인 화면

```text
[대기 요청]
  요청 금액 / 사유 / 상환 방식 / 예상 차감
[승인 상태 보드]
  대기 / 부분 상환 / 완료
```

### 월간 리포트 화면

```text
[월 선택]
[핵심 수치]
  총 지급 / 총 소비 / 총 저축 / 총 이자
[차트형 비율 카드]
[행동 달성률]
[코칭 인사이트]
```

## 4. 구현 순서

1. SQL 스키마 적용
2. Supabase 프로젝트 연결
3. Auth와 프로필/자녀 기본 CRUD 연결
4. 행동 규칙 및 로그 저장 연결
5. 거래 생성 로직 서버 함수화
6. 대시보드와 리포트 조회 연결
7. 승인 플로우 연결

## 5. 초보자 실행 순서

1. `npm install`
2. `.env.local` 생성
3. Supabase에서 새 프로젝트 생성
4. `supabase/schema.sql` 전체 실행
5. `npm run dev`
6. 현재 UI와 더미 데이터를 확인
7. 그 다음 더미 데이터 함수 대신 Supabase 쿼리로 교체
